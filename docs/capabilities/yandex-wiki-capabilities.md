## Краткое описание

Документ фиксирует, какие возможности Yandex Wiki будут доступны через MCP-сервер. Это решение по составу инструментов для работы со страницами базы знаний и динамическими таблицами (grids).

## Для кого этот документ

Документ рассчитан на смешанную аудиторию: разработчиков сервера и пользователей, которым важно понимать, что агент сможет делать с Wiki. Технические детали вынесены в таблицы.

## Зачем это нужно

Yandex Wiki предоставляет только REST API. MCP-сервер превращает методы API в инструменты, чтобы ИИ-агент мог искать, читать и редактировать страницы, работать с таблицами, добавлять комментарии и вложения на естественном языке.

## Основные понятия


| Понятие                               | Простое объяснение                                                          |
| ------------------------------------- | --------------------------------------------------------------------------- |
| Page (страница)                       | Документ в Wiki с заголовком и содержимым в формате разметки                |
| Slug                                  | Человекочитаемый адрес страницы, например `team/onboarding`                 |
| Page ID                               | Числовой идентификатор страницы                                             |
| Grid (динамическая таблица)           | Таблица данных, прикреплённая к странице, со строками, столбцами и ячейками |
| Resource (ресурс)                     | Объект, прикреплённый к странице: вложение или таблица                      |
| Recovery token (токен восстановления) | Код, который возвращается при удалении страницы и позволяет её восстановить |
| Descendants (потомки)                 | Дерево вложенных страниц под выбранной страницей                            |


## Как это работает

Все инструменты обращаются к REST API Wiki по базовому адресу `https://api.wiki.yandex.net/v1/`. В каждый запрос сервер добавляет заголовки:

- `Authorization: OAuth <токен>` — токен OAuth 2.0;
- `X-Org-Id` или `X-Cloud-Org-Id` — идентификатор организации.

Запросы выполняются от имени пользователя, которому выдан токен. Права в API совпадают с правами этого пользователя в интерфейсе Wiki: если страница недоступна для редактирования, соответствующий инструмент вернёт ошибку доступа.

## Принципы отбора возможностей

1. Покрываем все публичные методы API Wiki (по официальной документации и типизированным клиентам — 21 endpoint).
2. Делим инструменты на чтение и изменение; изменяющие можно отключить флагом `READ_ONLY`.
3. Реализацию ведём по приоритетам: сначала чтение и базовая работа со страницами, затем таблицы и вложения.
4. Удаление страницы возвращает токен восстановления — отдельный инструмент позволяет вернуть страницу.

## Соглашения об именовании инструментов

- Префикс группы: `wiki_`.
- Формат: `wiki_<сущность>_<действие>`, например `wiki_page_get`, `wiki_grid_add_rows`.
- Имена даются на английском в стиле `snake_case`.

## Состав инструментов по группам

### Группа page attachments — вложения страниц

Загрузка бинарных файлов входит в объём MVP. Она выполняется через сессию загрузки по частям (chunked upload), каждая часть — до 5 МБ. Инструмент `wiki_page_attachment_upload` инкапсулирует весь процесс: создаёт сессию, отправляет части, завершает сессию и прикрепляет файл к странице. Низкоуровневые шаги сессии описаны ниже как отдельные инструменты на случай ручного управления загрузкой.

| Инструмент                          | Действие                                              | Метод и endpoint                                       | Тип | Приоритет |
| ----------------------------------- | ----------------------------------------------------- | ------------------------------------------------------ | --- | --------- |
| `wiki_page_attachments_list`        | Получить вложения страницы                            | `GET /v1/pages/{id}/attachments`                       | R   | 4         |
| `wiki_page_attachment_upload`       | Загрузить локальный файл и прикрепить к странице (составной инструмент) | `POST /v1/upload_sessions` → `PUT .../upload_part` → `POST .../finish` → `POST /v1/pages/{id}/attachments` | W | 4 |
| `wiki_upload_session_create`        | Создать сессию загрузки файла                         | `POST /v1/upload_sessions`                             | W   | 5         |
| `wiki_upload_session_upload_part`   | Загрузить часть файла (до 5 МБ, `application/octet-stream`) | `PUT /v1/upload_sessions/{session_id}/upload_part?part_number=N` | W | 5 |
| `wiki_upload_session_finish`        | Завершить сессию загрузки                             | `POST /v1/upload_sessions/{session_id}/finish`         | W   | 5         |
| `wiki_page_attachment_attach`       | Прикрепить завершённую сессию к странице              | `POST /v1/pages/{id}/attachments` с `{upload_sessions: [...]}` | W | 5 |

## Формат содержимого страниц

Содержимое страниц передаётся и принимается в формате **Markdown**. Это относится к полям `content` при создании (`wiki_page_create`), обновлении (`wiki_page_update`) и дополнении (`wiki_page_append_content`).

Дополнительно Wiki поддерживает специальную разметку для встраивания вложенных файлов. После загрузки файла его можно вставить в текст страницы конструкцией:

```text
{% file src="<адрес файла>" name="<имя файла>" %}
```

Дописывание содержимого через `wiki_page_append_content` поддерживает указание места вставки в поле `body.location`: `bottom` (в конец), `top` (в начало) или к якорю.

## Что вынесено за рамки


| Возможность                            | Причина                                                                  |
| -------------------------------------- | ------------------------------------------------------------------------ |
| Авторизация по IAM-токену              | По решению заказчика используется OAuth 2.0 device flow                  |
| Управление правами доступа к страницам | API публично не документирует это как отдельный набор методов для агента |
| Старый редактор динамических таблиц    | Используем актуальный API grids                                          |


## Ограничения и важные условия

- Действия ограничены правами пользователя, которому выдан OAuth-токен.
- Содержимое страниц передаётся в формате **Markdown**; агент должен формировать корректную Markdown-разметку при создании и обновлении.
- Загрузка вложений выполняется через сессию загрузки по частям (chunked), размер части — до 5 МБ. Части отправляются с `Content-Type: application/octet-stream`.
- Удаление страницы возвращает `recovery_token`. Восстановление выполняется по токену через `POST /v1/recovery_tokens/{token}/recover`, поэтому токен нужно сохранять и возвращать пользователю.
- Получение потомков выполняется по `slug` через `GET /v1/pages/descendants`, а не по идентификатору страницы.
- Все изменяющие инструменты блокируются при включённом флаге `READ_ONLY`.

## Открытые вопросы


Ранее открытые вопросы закрыты по решению заказчика:

- Точные имена endpoint для `recover`, `descendants`, `comments`, `attachments` — уточнены по официальным примерам API и зафиксированы в таблицах выше.
- Загрузка бинарных вложений — входит в объём MVP (реализуется через сессию загрузки по частям).
- Формат содержимого страниц — Markdown.

| Вопрос                                                       | Почему важно уточнить                                            | Кто может ответить        |
| ------------------------------------------------------------ | ---------------------------------------------------------------- | ------------------------- |
| Поддерживаемые места вставки `body.location` (`top`/`bottom`/якорь) | Влияет на параметры инструмента `wiki_page_append_content`       | Тестирование на реальном API |
| Способ передачи файла в `wiki_page_attachment_upload` (путь в томе или содержимое в base64) | Влияет на сигнатуру инструмента загрузки                         | Заказчик                  |
