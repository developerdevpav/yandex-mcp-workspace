## Краткое описание

Документ фиксирует, какие возможности Yandex Tracker будут доступны через MCP-сервер в виде инструментов (tools). Это решение по составу функциональности: что именно сервер умеет делать, на какие методы REST API это опирается и как инструменты сгруппированы.

Документ — основа для реализации. Каждый инструмент в таблицах ниже соответствует одному вызову, который сможет выполнить ИИ-агент через MCP.

## Для кого этот документ

Документ рассчитан на смешанную аудиторию: разработчиков сервера, аналитиков и пользователей, которым важно понимать границы автоматизации. Технические детали (методы API, заголовки) вынесены в таблицы, а смысл каждой группы объясняется обычным текстом.

## Зачем это нужно

Yandex Tracker предоставляет только REST API. Чтобы ИИ-агент мог работать с задачами на естественном языке (создавать, искать, комментировать, переводить по статусам), нужен промежуточный слой — MCP-сервер, который превращает действия API в понятные инструменты с описанием и схемой параметров.

Этот документ отвечает на вопрос: «Что именно агент сможет делать с Tracker».

## Основные понятия


| Понятие                         | Простое объяснение                                                          |
| ------------------------------- | --------------------------------------------------------------------------- |
| Issue (задача)                  | Основная единица работы в Tracker: задача, баг, история                     |
| Queue (очередь)                 | Контейнер для задач одного проекта или команды, например `TREK`             |
| Issue key (ключ задачи)         | Человекочитаемый идентификатор задачи, например `TREK-42`                   |
| Transition (переход)            | Перевод задачи из одного статуса в другой по правилам рабочего процесса     |
| Worklog (учёт времени)          | Запись о потраченном на задачу времени                                      |
| Checklist (чек-лист)            | Список подпунктов внутри задачи                                             |
| Board (доска)                   | Визуальное представление задач по колонкам                                  |
| Sprint (спринт)                 | Ограниченный по времени отрезок работы на доске                             |
| Entity (сущность)               | Объект нового API: проект (`project`) или портфель (`portfolio`)            |
| Bulk change (массовая операция) | Одновременное изменение многих задач одной командой                         |
| Version (версия задачи)         | Счётчик изменений задачи; нужен для защиты от конфликтов при редактировании |


## Как это работает

Все инструменты обращаются к REST API Tracker по базовому адресу `https://api.tracker.yandex.net/v3/`. В каждый запрос сервер добавляет заголовки:

- `Authorization: OAuth <токен>` — токен, полученный через OAuth 2.0;
- `X-Org-ID` или `X-Cloud-Org-ID` — идентификатор организации (Яндекс 360 или Yandex Cloud соответственно).

Агент вызывает инструмент с понятными параметрами (например, ключ задачи и текст комментария). Сервер формирует HTTP-запрос, отправляет его в Tracker, разбирает ответ и возвращает агенту структурированный результат. Ошибки API (нет прав, не найдено, конфликт версий, превышен лимит запросов) сервер переводит в понятные сообщения.

## Принципы отбора возможностей

1. Покрываем максимум публичных методов API Tracker, как просил заказчик.
2. Делим инструменты на две категории: чтение (read) и изменение (write). Изменяющие инструменты можно глобально отключить флагом `READ_ONLY`.
3. Реализацию ведём по приоритетам: сначала чтение и работа с задачами, затем изменения и вспомогательные справочники.
4. Не дублируем функциональность: один инструмент — одно понятное действие.
5. Для пагинации и фильтрации пробрасываем параметры API без искажения смысла.

## Соглашения об именовании инструментов

- Префикс группы: `tracker_`.
- Формат: `tracker_<сущность>_<действие>`, например `tracker_issue_get`, `tracker_comment_add`.
- Действия: `get`, `list`, `search`, `count`, `create`, `update`, `delete`, плюс специальные (`transition`, `move`).
- Имена даются на английском в стиле `snake_case`, потому что это идентификаторы инструментов, а не текст для чтения.

## Что вынесено за рамки


| Возможность                         | Причина                                                              |
| ----------------------------------- | -------------------------------------------------------------------- |
| Триггеры и автодействия очередей    | Сложная конфигурация, редко нужна агенту; кандидат на будущую версию |
| Макросы                             | Зависят от интерфейсных сценариев; добавим позже при спросе          |
| Импорт задач (`/v3/issues/_import`) | Опасная операция с историей; требует отдельного решения              |
| Авторизация по IAM-токену           | По решению заказчика используется OAuth 2.0 device flow              |
| Вебхуки и подписки                  | Tracker не предоставляет это как часть REST API для агента           |


## Ограничения и важные условия

- Доступные действия ограничены правами пользователя, от имени которого выдан OAuth-токен. Если у пользователя нет прав на действие — API вернёт `403`.
- При изменении задачи Tracker использует поле `version` для защиты от конфликтов. Инструмент `tracker_issue_update` поддержит передачу версии; при конфликте вернётся понятная ошибка `409`/`412`.
- Пагинация: по умолчанию 50 объектов на страницу, управляется параметрами `perPage` и `page`.
- Лимиты запросов: при `429` сервер выполнит повтор с экспоненциальной задержкой.
- Все изменяющие инструменты блокируются при включённом флаге `READ_ONLY`.

## Открытые вопросы


Загрузка и скачивание бинарных вложений входят в объём MVP (по решению заказчика). Загрузка вложений в задачу выполняется методом `POST /v3/issues/{key}/attachments` (multipart-форма с файлом), временные вложения для создания задачи — методом `POST /v3/attachments`.

| Вопрос                                                               | Почему важно уточнить                         | Кто может ответить |
| -------------------------------------------------------------------- | --------------------------------------------- | ------------------ |
| Нужны ли инструменты триггеров и автодействий                        | Влияет на объём первой версии                 | Заказчик           |
| Какой тип организации используется (`X-Org-ID` или `X-Cloud-Org-ID`) | Определяет заголовок авторизации по умолчанию | Заказчик           |
